## What is ARIA?

ARIA is an AI-powered research tool built around one principle: **good research means being willing to be wrong.**

You give it a research question — "Is NVDA overvalued?", "Will small-cap value outperform growth in 2026?" — and it runs a structured process that forces challenge at every step. Counter-evidence is mandatory, not optional, failure conditions must be named and specific, and confidence ratings constrain the language of the conclusion. The result is a structured research memo built to survive scrutiny.

**Why I built it:** Before starting my first full-time job after college, I used to do my own investment research. I simply ran out of time and wanted to make a system that produced something automatic and reliable. I also noticed that most AI tools make confirmation bias worse, not better: If you think something is a good investment, an LLM can pick up on that vibe and just tell you that it is. ARIA inverts that by making counter-evidence mandatory and encoding falsification discipline directly into the tool layer, not just the prompt.

**Future vision:** After a research run, ARIA won't go quiet. A monitoring loop periodically re-evaluates your active theses against live market data and fresh headlines, alerting you when a named failure condition has triggered. And when you eventually close a thesis, you record whether it turned out correct — accumulating a calibration record of how well your research (and the model's stated confidence) predicted what actually happened.

**You don't need to be a developer to use ARIA.** You need Python and either a local AI model or a cloud API key.

## The research philosophy

Good research is a discipline, not a tool. Most AI assistants optimize for giving you an answer you'll be satisfied with, but this project treats every question as a hypothesis that must survive hostile examination:

- **One directional claim.** You can't generate a "bull/base/bear" scenario matrix. You must commit to one claim that can be proven wrong.
- **Mandatory counter-evidence.** Every memo requires a counter-evidence section before it can be finalized. The model cannot skip it.
- **Specific failure conditions.** Vague risks ("macro headwinds") are blocked. Failure conditions must be named, measurable, and monitorable. Not "rates could rise", but "10Y Treasury yield exceeds 5.5% for two consecutive weeks."
- **Confidence that means something.** Medium or Low confidence automatically constrains the conclusion language. Research uncertainty must flow through to the recommendation. You can't be hedged in your analysis and confident in your call.
- **Future vision: Ongoing accountability.** Theses stay open until you close them. Failure conditions will be re-evaluated against live data on a schedule. When conditions trigger, you're alerted. When enough time passes, you record the outcome and the database accumulates a calibration record: what percentage of High-confidence calls were correct?

## Requirements

**Python 3.11+**

```bash
pip install -e .
```

**A function-calling-capable AI model.** You have two options:

- **Local model**: Load a model in [LM Studio](https://lmstudio.ai) or Ollama and start the local server (default port 1234). No API key or internet required — your data stays on your machine.
- **Cloud API**: Use Anthropic or OpenAI. Set `ANTHROPIC_API_KEY` or `OPENAI_API_KEY` in your environment.

---

## Configuration

All settings live in `aria_config.yaml`.

**Local model (LM Studio / Ollama):**
```yaml
model:
  local: qwen2.5-14b-instruct   # match the model name in LM Studio
  local_base_url: http://localhost:1234/v1
  local_api_key: lm-studio
  routing: local
```

This has been tested on local context window lengths of 8192; lower may result in incomplete memos.

**Cloud API (Anthropic / OpenAI):**
```yaml
model:
  default: claude-sonnet-4-6    # or gpt-4o, etc.
  routing: api
# Set ANTHROPIC_API_KEY or OPENAI_API_KEY in your environment
```

**Privacy mode** — forces local-only routing regardless of task type:
```yaml
model:
  privacy_mode: true
```

## Usage

### Run a research query

```bash
aria "Is NVDA overvalued relative to the semiconductor sector?"
```

By default ARIA runs in **checkpoint mode**: it sharpens the research question, you confirm or revise it, then the agentic loop runs the full analysis.

To run without the confirmation step:
```bash
aria "your query" --mode autonomous
```

Other options:
```
--config path/to/aria_config.yaml   # use a different config file
--save output.md                    # save the memo to a specific path
--no-stream                         # print full response at the end
```

Memos are auto-saved to `memos/YYYY-MM-DD_<id[:8]>.md` on every successful run.

---

### Record an outcome

After a run ARIA prints a session ID. Come back later and record whether the thesis held up:

```bash
aria outcome a1b2c3d4-... --result correct|incorrect|partial
aria outcome a1b2c3d4-... --result partial --note "Capex held but margins compressed"
```

The command shows the original thesis and failure conditions before writing, so you know exactly what you're rating.

---

### Browse and manage sessions

```bash
# Detailed, fiterable table
aria db list
aria db list --ticker NVDA
aria db list --status challenged          # active | challenged | resolved
aria db list --since 2026-01-01
aria db list --outcome pending            # pending | correct | incorrect | partial

# Full memo view: all sections, sources, and saved memo path
aria db show a1b2c3d4                     # prefix match (≥8 chars)
aria db show a1b2c3d4-e5f6-...            # or full UUID

# Delete a session and all related data
aria db delete a1b2c3d4                   # confirms before deleting
aria db delete a1b2c3d4 --force           # skip confirmation

# Export to JSON or CSV
aria db export                            # JSON to stdout
aria db export --format csv --output sessions.csv

# Summary panel
aria db stats
```

The legacy `aria history` command remains available and unchanged.

### Thesis monitoring

ARIA re-evaluates active theses on a schedule against live price data and recent headlines, alerting when a named failure condition triggers.

```bash
# Check all active theses now
aria monitor run

# Check a single session
aria monitor run --session a1b2c3d4-...

# Start a background scheduler (blocks; Ctrl-C to stop)
aria monitor start

# Browse past monitor runs
aria monitor history
aria monitor history --session a1b2c3d4-... --limit 10
```

**How it works:** For each active session the monitor (1) fetches 3-month price history for the detected ticker, (2) runs a targeted news search, then (3) asks the model whether any named failure condition has triggered. Results are written to `monitor_runs`. If a thesis is challenged, `thesis_status` updates and, if configured, a Discord alert fires.

**Configure Discord alerts** in `aria_config.yaml`:
```yaml
monitor:
  interval_hours: 24
  notify_on: challenged       # "challenged" (default) or "all"
  discord_webhook: https://discord.com/api/webhooks/...
```

## Architecture

**Falsification is enforced at the tool layer, not the prompt layer.** The model cannot call `finalize_memo` without first completing a `counter_thesis` search and passing 8 quality gates in `tool_executor.py` — this is mechanical enforcement, not prompt engineering. Counter-evidence is not optional.

**Local-first by design.** ARIA runs fully on-device with models like Qwen via LM Studio or Ollama (tested on Apple Silicon). No research data, queries, or memos leave your machine unless you explicitly configure a cloud API. This matters for finance use cases where query content is itself sensitive.

**SQLite as session state.** Every run — thesis, sources, failure conditions, confidence, outcome — is persisted to a local SQLite database. The monitoring loop reads from this state on each check, so context never drifts between runs. The database is the source of truth; the LLM is stateless.

│   ├── core.py           # AgentLoop — orchestrates the agentic tool loop
│   ├── tool_schemas.py   # OpenAI-compatible tool definitions (web_search,
│   │                     #   get_financial_data, store_finding, finalize_memo)
│   └── tool_executor.py  # Dispatches tool calls; enforces thesis quality gates
├── cli/
│   ├── main.py           # Entrypoint — routes research/outcome/history/monitor/db
│   └── db_commands.py    # aria db list/show/delete/export/stats
├── config/
│   └── loader.py         # YAML config with env-var expansion
├── models/

### Quality gates in the agentic loop

`tool_executor.py` enforces research discipline at the tool level — the model cannot finalize a memo that violates these rules:

| Gate | What it blocks |
|---|---|
| **Thesis directional** | No simultaneous bull/base/bear framing; the thesis must be one testable claim |
| **Uniqueness** | Asserting a "unique" financial fact without naming a specific competitor and citing evidence they lack it |
| **Confidence propagation** | Medium or Low confidence that doesn't produce conditional language in the conclusion |
| **Section overwrite** | Re-writing a stored section — each section is write-once |
| **Cross-asset search** | Skipping `web_search(purpose="cross_asset")` when the query references two or more assets |
| **Finalize completeness** | Calling `finalize_memo` before all 8 sections are stored |
| **Domain validation** | Citing a source domain that wasn't actually searched |
| **Price grounding** | Failing to anchor claims to fetched price data |

### How the agentic loop runs

1. Model receives the query and tool schemas via `bind_tools()`
2. Model calls `web_search(purpose="background")` — up to 2 searches, orient
3. Model calls `get_financial_data()` for each relevant ticker (up to 4 calls)
4. Model calls `web_search(purpose="cross_asset")` if multiple assets are referenced (mandatory)
5. Model calls `web_search(purpose="pro_thesis")` — evidence for the claim
6. Model calls `web_search(purpose="counter_thesis")` — evidence against
7. Model calls `store_finding()` once per section, passing each quality gate
8. Model calls `finalize_memo()` — validated, rendered to markdown, saved to SQLite

Each search purpose is capped at 3 calls. After the cap, `store_finding` is the only available path.

If the model stalls (3 consecutive steps with no tool calls), the loop falls back gracefully: whatever text the model produced is returned with `fallback_used: true`. Fallback runs are not saved to the database.

## Session database

Every successful run is written to `aria_research.db` (SQLite, path configurable via `agent.db_path`).

```sql
sessions      — query, ticker, all 8 memo sections, model metadata, timestamp, thesis_status
sources       — url, title, purpose (background/pro_thesis/counter_thesis) per session
outcomes      — result (correct/incorrect/partial), note, evaluated_at per session
monitor_runs  — status (ok/challenged/error), summary, checked_at per session
```

Use the `aria db` commands for interactive management, or query directly:

```sql
-- Calibration: accuracy by model
SELECT model_name,
       COUNT(*) AS total,
       SUM(CASE WHEN o.result = 'correct' THEN 1 ELSE 0 END) AS correct,
       printf('%.0f%%', 100.0 * SUM(CASE WHEN o.result = 'correct' THEN 1 ELSE 0 END) / COUNT(*)) AS accuracy
FROM sessions s
INNER JOIN outcomes o ON s.id = o.session_id
GROUP BY model_name;

-- Open theses with failure conditions — candidates for monitoring
SELECT id, created_at, ticker, thesis, failure_conditions
FROM sessions
WHERE id NOT IN (SELECT session_id FROM outcomes)
  AND failure_conditions IS NOT NULL
ORDER BY created_at ASC;
```
